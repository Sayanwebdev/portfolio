Why Does My Portfolio Have a Login Form? (It's Not What You Think)

Published: 12 November 2025 ‚Ä¢ Sayan Mondal

If you've browsed my portfolio, you may have noticed the "Test Magic Link Login" form and wondered, "A login... for a static portfolio? Why?" That's a fair question. My portfolio has no private content, no members-only area, and no store ‚Äî so why build a high-tech, secure login system for it?

The short answer: it's a tech demo. It's like installing a bank-vault door on a garden shed. The shed doesn't need the security, but building and installing the vault door proves you know how to.

What is a "Magic Link" anyway?

A magic link is a form of passwordless authentication you've likely seen on services like Slack or Medium. You enter your email address, the system sends a secure one-time link to your inbox, and when you click that link you're logged in ‚Äî no password required. It's a clean, modern user experience, and I chose to build it as the demo for this site.

The challenge: adding a backend to a static site

A static site (HTML/CSS/JS) has no server-side components. To implement login you usually need:

  • a database to store user emails
  • a server to process login requests
  • a secure way to create and verify one-time tokens
  • an email service to deliver the magic link

Building all of that from scratch is a big project. Instead, I used a Backend-as-a-Service (BaaS).

The solution: Backend-as-a-Service (Supabase)

I used Supabase, an open-source alternative to Firebase. Supabase provides the backend pieces I needed in one package, so I didn't have to build an entire auth system from the ground up. At a high level the steps were:

  1. Create a Supabase project and enable Email Authentication with Magic Link.
  2. Add the supabase-js client to my site.
  3. Write a small frontend function to call Supabase's auth API.

The core JavaScript (simplified)

One short function does the work of requesting a magic link from Supabase:

const { data, error } = await supabase.auth.signInWithOtp({
  email: userEmail,
  options: { emailRedirectTo: window.location.origin }
});

Supabase handles token generation, email delivery, and verification. My frontend simply captures the email and calls that API.

So ‚Äî what's the point?

This little feature is a trophy that demonstrates several skills:

  • API integration ‚Äî reading docs and wiring a third-party service.
  • Full-stack awareness ‚Äî understanding authentication flows and sessions.
  • Modern JavaScript ‚Äî async/await and promise handling.
  • Problem-solving ‚Äî identifying a gap (no backend) and using a modern, scalable solution (BaaS).

What's next?

Right now, the vault door sits on the shed. The next step is to build something worth protecting ‚Äî for example, a real-time comments section or a live guestbook. Logged-in users could write to the Supabase database and others would see messages appear in real time.

If you'd like to try the login yourself, go back to the homepage and use the "Test Magic Link Login" form. It's a demo ‚Äî please don't use production account credentials when testing.

Building a Prepaid Energy Meter using ESP32 ‚ö°

Published: October 2025

In this project, I developed a smart prepaid energy metering system using the ESP32 microcontroller. The system accurately measures real-time power consumption with the help of current and voltage sensors and automatically deducts credit based on energy usage. Once the prepaid balance reaches zero, the power supply is automatically disconnected to prevent overuse.

To enhance functionality, I integrated IoT connectivity that allows users to monitor their energy consumption and remaining balance through a cloud dashboard. The system also supports online recharge features and real-time alerts via Firebase or Blynk platforms.

This project demonstrates how IoT and automation can make energy billing more transparent, efficient, and user-friendly, particularly in smart home and industrial applications.
